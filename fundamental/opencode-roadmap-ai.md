# BAGIAN II: AI ENGINEERING - ROADMAP JALUR CEPAT PAHAM

## 2.1 Definisi AI Engineering

### Apa Itu AI Engineering?

**AI Engineering** adalah disiplin yang fokus pada **pembangunan, deployment, dan maintenance** sistem AI dalam lingkungan production. AI Engineer bukan researcher yang membuat model baru, tapi application builder yang mengintegrasikan model-model yang sudah ada (pre-trained) ke dalam sistem nyata.

### Perbandingan: AI Engineer vs Data Scientist vs ML Engineer

| Aspek | AI Engineer | Data Scientist | ML Engineer |
|-------|-------------|----------------|-------------|
| **Fokus Utama** | Membangun aplikasi berbasis AI | Analisis data & insight | Training & deployment model |
| **Pendekatan** | Integration & System Design | Statistical Analysis | Model Development |
| **Skill Utama** | Python, APIs, RAG, Deployment | Statistics, SQL, Visualization | TensorFlow, PyTorch, Training |
| **Output** | AI-powered applications | Reports, predictions | Trained models |

### Apa yang Dilakukan AI Engineer Setiap Hari?

1. **Building LLM Applications** - Membuat chatbot, virtual assistant, content generator
2. **Designing RAG Pipelines** - Menghubungkan LLM dengan data internal perusahaan
3. **Building AI Agents** - Sistem yang bisa autonomous menjalankan tugas
4. **Implementing Evaluation Systems** - Mengukur kualitas output AI
5. **Optimizing for Production** - Memastikan sistem AI berjalan reliable dan cost-effective

### Tahap 1: Fondasi Programming (Bulan 1-2)

#### Mengapa Fondasi Programming Sangat Penting?

Ini adalah tahap yang **paling sering dilewati** oleh pemula, tapi justru ini yang **paling menentukan** keberhasilan mereka selanjutnya. Tanpa fondasi programming yang kuat, Anda akan:

- Kesulitan memahami code examples
- Tidak bisa debug saat terjadi error
- Sulit mengintegrasikan berbagai tools
- Akan stuck di tutorial hell selamanya

#### Topik yang Harus Dikuasai

**1.1 Python Fundamentals**

```python
# Yang harus Anda kuasai di Python:

# Variabel dan Tipe Data
nama = "Budi"           # String
umur = 25               # Integer
tinggi = 170.5         # Float
is_mahasiswa = True     # Boolean

# List dan Dictionary
mahasiswa = {
    "nama": "Budi",
    "jurusan": "Informatika",
    "semester": 5,
    "nilai": [85, 90, 78, 92]
}

# Control Flow
if umur >= 18:
    print("Dewasa")
else:
    print("Belum dewasa")

# Loops
for nilai in mahasiswa["nilai"]:
    print(nilai)

# Functions
def hitung_rata_rata(nilai_list):
    return sum(nilai_list) / len(nilai_list)

# List Comprehension
nilai_atas_80 = [n for n in mahasiswa["nilai"] if n > 80]

# Error Handling
try:
    result = 10 / 0
except ZeroDivisionError:
    print("Tidak bisa dibagi dengan nol")
```

**1.2 Git & GitHub**

Git adalah sistem versioning yang Wajib dikuasai. Bayangkan Anda sedang bekerja dalam tim yang terdiri dari 10 orang. Bagaimana Anda melacak perubahan kode dari setiap orang? Bagaimana jika ada yang merusak kode yang sudah berjalan? Git menjawab semua masalah ini.

```bash
# Perintah Git yang paling sering digunakan

# Inisialisasi repo baru
git init

# Clone repo yang sudah ada
git clone https://github.com/user/repo.git

# Cek status file
git status

# Menambahkan file ke staging
git add nama_file.txt
git add .                    # Menambahkan semua file

# Commit perubahan
git commit -m "Menambahkan fitur baru"

# Melihat history commit
git log

# Membuat branch baru
git branch fitur-baru

# Pindah ke branch lain
git checkout fitur-baru
git switch fitur-baru       # Cara baru

# Merge branch
git merge fitur-baru

# Push ke remote
git push origin main

# Pull dari remote
git pull origin main
```

##### Breakdown Detail: Git & GitHub

**Basic Commands (Wajib Hapal):**
- [ ] `git init` - Initialize repository baru
- [ ] `git clone <url>` - Clone repository existente
- [ ] `git status` - Cek status working directory
- [ ] `git add <file>` - Stage perubahan
- [ ] `git commit -m "message"` - Commit dengan pesan
- [ ] `git log` - Lihat history commit
- [ ] `git diff` - Lihat perubahan sebelum commit
- [ ] `git push` - Push ke remote
- [ ] `git pull` - Pull dari remote
- [ ] `git fetch` - Fetch tanpa merge

**Branching:**
- [ ] `git branch` - List semua branch
- [ ] `git branch <nama>` - Buat branch baru
- [ ] `git checkout <branch>` - Pindah ke branch
- [ ] `git switch <branch>` - Cara modern pindah branch
- [ ] `git checkout -b <branch>` - Buat dan pindah sekaligus
- [ ] `git branch -d <branch>` - Hapus branch
- [ ] `git merge <branch>` - Merge branch ke current
- [ ] `git rebase <branch>` - Rebase current branch ke branch lain

**Remote Operations:**
- [ ] `git remote -v` - List remote repositories
- [ ] `git remote add origin <url>` - Tambah remote
- [ ] `git push -u origin <branch>` - Push dengan upstream tracking
- [ ] `git push origin --delete <branch>` - Hapus remote branch
- [ ] `git pull --rebase` - Pull dengan rebase strategy

**Advanced Git:**
- [ ] `git stash` - Simpan perubahan sementara
- [ ] `git stash pop` - Kembalikan perubahan stash
- [ ] `git cherry-pick <commit>` - Apply commit tertentu
- [ ] `git reset --soft/mixed/hard` - Reset commits
- [ ] `git revert <commit>` - Revert commit tertentu
- [ ] `git tag <version>` - Buat release tag
- [ ] `git reflog` - Recovery dari mistakes

**Merge Conflicts:**
- [ ] Identifikasi conflict: `git status` akan show `both modified`
- [ ] Buka file dan cari `<<<<<<< HEAD`, `=======`, `>>>>>>>`
- [ ] Resolve manually dengan memilih perubahan yang benar
- [ ] `git add <file>` setelah resolve
- [ ] `git commit` untuk complete merge

**GitHub Specific:**
- [ ] Pull Requests (PR) - Request untuk merge perubahan
- [ ] Code Reviews - Review perubahan sebelum merge
- [ ] Fork - Copy repository orang lain ke akun sendiri
- [ ] Issues - Track bugs, features, tasks

**1.3 Data Structures & Algorithms**

Ini adalah fondasi computational thinking. Anda tidak perlu jadi expert algoritma seperti di competitive programming, tapi Anda harus memahami:

```
Data Structures yang harus dipahami:
├── Lists/Arrays - Koleksi data terurut
│   ├── O(1) access by index
│   ├── O(n) search
│   └── O(1) append, O(n) insert
│
├── Dictionaries/HashMaps - Key-value pairs
│   ├── O(1) lookup average
│   └── Sangat efisien untuk pencarian
│
├── Sets - Koleksi unik tanpa duplikat
│   ├── O(1) lookup average
│   └── Berguna untuk eliminasi duplikat
│
└── Stacks & Queues - FIFO dan LIFO
    ├── Stack: like pile of plates
    └── Queue: like waiting line
```

##### Breakdown Detail: Data Structures & Algorithms

**Arrays/Lists:**
- [ ] Definition: Ordered, indexed collection
- [ ] Time Complexity:
  - Access by index: O(1)
  - Search: O(n)
  - Append: O(1) amortized
  - Insert at index: O(n)
  - Delete at index: O(n)
- [ ] Python: `list` adalah dynamic array
- [ ] Common Operations:
  - `append()`, `pop()` - O(1)
  - `insert()`, `remove()` - O(n)
  - `index()`, `count()` - O(n)
  - `sort()`, `reverse()` - O(n log n)
- [ ] Use cases: Sequential data, ordered collections, stack/queue implementation

**Linked Lists:**
- [ ] Definition: Nodes with data + pointer to next node
- [ ] Time Complexity:
  - Access: O(n)
  - Insert at head: O(1)
  - Insert at tail: O(1) with tail pointer
  - Delete: O(1) if node reference given
- [ ] Types: Singly linked, Doubly linked, Circular
- [ ] Python: Not built-in, but can implement manually
- [ ] Use cases: Frequent insert/delete at ends, memory efficiency

**Stacks:**
- [ ] Definition: LIFO (Last In, First Out)
- [ ] Operations: `push()` (top), `pop()` (top), `peek()` (top)
- [ ] Time Complexity: All O(1)
- [ ] Python: Use `list` with `append()` and `pop()`
- [ ] Use cases: Undo mechanisms, expression evaluation, DFS, function calls

**Queues:**
- [ ] Definition: FIFO (First In, First Out)
- [ ] Operations: `enqueue()` (rear), `dequeue()` (front)
- [ ] Time Complexity: All O(1) with linked list implementation
- [ ] Python: `collections.deque` for efficiency
- [ ] Types: Simple queue, Circular queue, Priority queue
- [ ] Use cases: BFS, task scheduling, print queue

**HashMaps/Dictionaries:**
- [ ] Definition: Key-value pairs dengan hash function
- [ ] Time Complexity:
  - Lookup/Insert/Delete: O(1) average, O(n) worst case
  - Collision handling: Chaining atau Open addressing
- [ ] Python: `dict` dengan open addressing
- [ ] Requirements: Keys must be hashable (immutable)
- [ ] Use cases: Caching, lookups, counting frequencies

**Sets:**
- [ ] Definition: Unordered collection of unique elements
- [ ] Operations: `add()`, `remove()`, `union()`, `intersection()`
- [ ] Time Complexity: O(1) average for add/remove/lookup
- [ ] Python: `set` dan `frozenset`
- [ ] Use cases: Deduplication, membership testing, set operations

**Trees:**
- [ ] Definition: Hierarchical structure dengan parent-child relationships
- [ ] Types:
  - Binary Tree: Max 2 children per node
  - Binary Search Tree (BST): Left < Root < Right
  - Balanced Trees: AVL, Red-Black Tree
  - N-ary Tree, Trie (prefix tree)
- [ ] Tree Traversals:
  - Inorder: Left → Root → Right
  - Preorder: Root → Left → Right
  - Postorder: Left → Right → Root
  - Level order: BFS by level
- [ ] Use cases: File systems, DOM, database indexes, expression trees

**Graphs:**
- [ ] Definition: Nodes (vertices) connected by edges
- [ ] Types:
  - Directed vs Undirected
  - Weighted vs Unweighted
  - Cyclic vs Acyclic
- [ ] Representations: Adjacency Matrix, Adjacency List
- [ ] Common Algorithms:
  - BFS: Shortest path unweighted
  - DFS: Path finding, cycle detection
  - Dijkstra: Shortest path weighted
  - Topological Sort: DAG ordering
- [ ] Use cases: Social networks, routing, dependencies

**Big O Notation (Wajib Hafal):**
```
O(1)       - Constant     - Array access
O(log n)   - Logarithmic  - Binary search
O(n)       - Linear       - Array search
O(n log n) - Linearithmic - Efficient sorting
O(n²)      - Quadratic    - Nested loops
O(2ⁿ)      - Exponential  - Recursive fibonacci
O(n!)      - Factorial    - Permutations
```

**Sorting Algorithms:**
- [ ] Bubble Sort: O(n²) - Simple but slow
- [ ] Selection Sort: O(n²) - Simple, minimal swaps
- [ ] Insertion Sort: O(n²) - Good for small/nearly sorted
- [ ] Merge Sort: O(n log n) - Stable, recursive
- [ ] Quick Sort: O(n log n) avg - Fast in practice
- [ ] Python: `sorted()` menggunakan Timsort O(n log n)

**Search Algorithms:**
- [ ] Linear Search: O(n) - Check each element
- [ ] Binary Search: O(log n) - Sorted arrays only
- [ ] Hash Search: O(1) - Using hash maps
- [ ] Tree Search: O(log n) for balanced trees

**Recursion:**
- [ ] Base case: When to stop
- [ ] Recursive case: Call itself
- [ ] Stack overflow risk for deep recursion
- [ ] Tail recursion optimization (not in Python)
- [ ] Can always convert to iteration

**Practical Algorithms for AI Engineer:**
- [ ] Hashing: Count frequencies, detect duplicates
- [ ] Two pointers: Pair finding, palindrome check
- [ ] Sliding window: Subarray problems, longest substring
- [ ] Divide and conquer: Merge sort, binary search variations
- [ ] Dynamic programming: Memoization, tabulation

**Indikator Proficiency:**
- [ ] Bisa implement basic data structures dari scratch
- [ ] Paham Big O dan bisa analyze complexity
- [ ] Tau kapan pakai data structure yang mana
- [ ] Bisa implement sorting dan searching algorithms
- [ ] Paham recursion dan iteration alternatives

**Common Pitfalls:**
- [ ] Jangan pakai list untuk frequent insert/delete di tengah
- [ ] Use set untuk membership testing, bukan list

- [ ] Graph representation affects algorithm choice

#### 2.2 LLM Intuition - Bagaimana LLM Bekerja

Ini adalah pengetahuan yang **SANGAT PENTING** untuk semua three disciplines. Anda harus benar-benar memahami konsep-konsep ini:

**Tokenization**

LLM tidak memahami kata-kata secara langsung. Mereka memahami **tokens**. Token adalah unit-unit kecil dari teks.

```python
# Contoh sederhana tokenization

# Kalimat: "Hello, world!"
# Mungkin di-tokenize menjadi: ["Hello", ",", " world", "!"]

# Implikasinya:
# - "Hello" = 1 token
# - "Hello, world!" = 4 tokens
# - "Hello world" = 3 tokens (tanpa tanda baca)

# Mengapa ini penting?
# - Context window dibatasi oleh JUMLAH TOKEN, bukan kata
# - Biaya API biasanya dihitung per token
# - Bahasa Indonesia cenderung memiliki token lebih banyak per kata
#   dibanding bahasa Inggris karena banyak partikel
```

**Context Window**

Context window adalah "memori kerja" dari LLM. Ini menentukan berapa banyak teks yang bisa diproses dalam satu request.

```
Context Window Comparison (2026):
├── GPT-4o: 128,000 tokens (~96 halaman buku)
├── Claude 3.5: 200,000 tokens (~150 halaman buku)
├── Gemini 2.0: 1 juta tokens (~750 halaman buku)
└── Model lokal: Bervariasi (biasanya 4K-32K)

Praktikal Implikasi:
- Anda TIDAK bisa memberikan seluruh dokumen 500 halaman dalam satu prompt
- Anda harus cerdas memilih "context" yang relevan
- RAG (Retrieval-Augmented Generation) ada untuk解决 masalah ini
```

**Temperature dan Sampling**

Temperature mengontrol seberapa "kreatif" atau "deterministic" output yang dihasilkan.

```python
# Analogi sederhana:

# Temperature = 0.0 (Deterministic)
# Like bertanya: "Berapa 2 + 2?"
# Jawaban hampir selalu: "4"
# Cocok untuk: Fakta, kode, instruksi yang harus presisi

# Temperature = 0.7 (Balanced)
# Like bertanya: "Ceritakan sebuah cerita lucu tentang kucing"
# Jawaban cukup bervariasi tapi masih masuk akal
# Cocok untuk: Conversational AI, content generation

# Temperature = 1.5 (Creative)
# Like bertanya: "Buatkan 10 ide startup yang aneh"
# Jawaban sangat bervariasi dan mungkin out-of-the-box
# WARNING: Bisa menghasilkan hallucination yang tinggi
# Cocok untuk: Brainstorming, creative writing
```

**Hallucination**

Hallucination adalah fenomena di mana LLM menghasilkan informasi yang **terdengar meyakinkan tapi sebenarnya salah**.

```python
# Contoh Hallucination:

User: "Siapa presiden Indonesia tahun 2024?"
LLM: "Presiden Indonesia tahun 2024 adalah Prabowo Subianto."

# Fakta ini BENAR (asumsi di waktu penulisan).
# Tapi bagaimana jika pertanyaan ini?

User: "Siapa penemu Python yang lahir tahun 1985?"
LLM: "Python diciptakan oleh Guido van Rossum pada tahun 1985 di Belanda."

# ERROR: 
# - Guido van Rossum memang pencipta Python
# - Tapi ia lahir pada 31 Januari 1956, bukan 1985
# - LLM "melampirkan" detail yang salah ke fakta yang benar

# Ini adalah HALLUCINATION - LLM menghasilkan informasi yang salah
# tapi formatnya sangat meyakinkan
```

Mengapa hallucination terjadi? Karena LLM adalah **next-token predictor**. Mereka memprediksi kata berikutnya berdasarkan probabilitas. Mereka TIDAK memiliki akses ke "truth" - mereka hanya tahu pola dalam data training.

**Mitigasi Hallucination:**
1. Gunakan RAG untuk grounding dengan data yang valid
2. Berikan constraints yang jelas dalam prompt
3. Gunakan evaluation untuk mendeteksi hallucination
4. Minta model untuk cite sources

##### Breakdown Detail: LLM Intuition

**Tokenization (Wajib Pahami):**
- [ ] Text → Tokens: LLM tidak baca kata, tapi token
- [ ] 1 token ≈ 4 karakter bahasa Inggris, 1-2 kata
- [ ] Bahasa Indonesia: Token lebih banyak karena banyak partikel
- [ ] API pricing: Per token, bukan per request
- [ ] Tools: OpenAI tokenizer, tiktoken library
- [ ] Max tokens: Context window limit

**Context Window (Wajib Hafal):**
- [ ] Maximum tokens yang bisa diproses dalam satu request
- [ ] Includes: prompt + output + system message
- [ ] Model populer 2026:
  - GPT-4o: 128K tokens
  - Claude 3.5/3.7: 200K tokens
  - Gemini 2.0: 1M tokens
  - Local models: Usually 4K-32K
- [ ] Strategies untuk context panjang:
  - Truncation (potong)
  - Summarization (ringkas)
  - Chunking (pecah)
  - RAG (retrieve)

**Temperature:**
- [ ] 0.0: Almost deterministic, sama setiap kali
- [ ] 0.3: Low creativity, focused
- [ ] 0.5-0.7: Balanced (recommended untuk general use)
- [ ] 0.8-1.0: Creative, varied output
- [ ] >1.0: Chaotic, unreliable
- [ ] Tips: Fakta/coding = 0-0.3, brainstorming = 0.7-0.9

**Top-K & Top-P Sampling:**
- [ ] Top-K: Pilih dari K token teratas
- [ ] Top-P (Nucleus): Pilih token sampai probabilitas kumulatif = P
- [ ] Top-P lebih flexible dari Top-K
- [ ] Typical values: top_p=0.9, top_k=40

**Model Types & Providers:**
- [ ] **GPT Series**: OpenAI, ChatGPT, API
  - GPT-4o: Latest, multimodal
  - GPT-4 Turbo: Faster, cheaper
  - GPT-3.5 Turbo: Budget option
- [ ] **Claude**: Anthropic
  - Claude 3.5 Sonnet: Best for coding
  - Claude 3 Haiku: Fast, cheap
- [ ] **Gemini**: Google
  - Gemini 2.0 Flash: 1M context, fast
  - Gemini Pro: Balanced
- [ ] **Open Source**: Ollama, LM Studio, HuggingFace
  - Llama 3, Mistral, Qwen
  - Bisa running lokal (privacy, offline)

**How LLMs Actually Work (High Level):**
1. **Input Processing**: Text → Tokens (tokenizer)
2. **Embedding**: Tokens → Vectors (embeddings)
3. **Processing**: Vectors → Through Transformer layers
4. **Output**: Probability distribution over next token
5. **Sampling**: Pick token based on temperature/distribution
6. **Repeat**: Until EOS token atau max tokens

**Key LLM Concepts:**
- [ ] **Attention Mechanism**: How tokens relate to each other
- [ ] **Transformers**: Architecture berbasis attention
- [ ] **Pre-training**: Learn language from massive text
- [ ] **Fine-tuning**: Adapt ke specific tasks
- [ ] **RLHF**: Reinforcement Learning from Human Feedback
- [ ] **Alignment**: Make model helpful, harmless, honest

**Hallucination - Deep Dive:**
- [ ] **Why**: LLM = next-token predictor, bukan fact retriever
- [ ] **Types**: Factual, logical, statistical
- [ ] **Mitigation**:
  - RAG (ground with real data)
  - Chain-of-thought (reason step by step)

- [ ] Assistant message: Previous responses
- [ ] Few-shot examples: Show, don't just tell
- [ ] Remember: Everything dalam context window

- [ ] Be explicit tentang format output
- [ ] Chain-of-thought untuk complex reasoning
- [ ] Test dengan edge cases

#### 3.1 RAG (Retrieval-Augmented Generation)

**Apa itu RAG?**

RAG adalah teknik untuk menghubungkan LLM dengan **data eksternal**. Tanpa RAG, LLM hanya bisa menjawab berdasarkan knowledge yang ada di training data-nya. Dengan RAG, LLM bisa menjawab pertanyaan tentang dokumen atau data yang ANDA miliki.

```
Arsitektur RAG Sederhana:

┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Dokumen    │────▶│  Chunker    │────▶│  Embedding  │
│  (Input)    │     │  (Pecah)    │     │  Model      │
└─────────────┘     └─────────────┘     └─────────────┘
                                            │
                                            ▼
                    ┌─────────────┐     ┌─────────────┐
                    │  Vector DB  │◀────│   Vectors   │
                    │  (Simpan)   │     │   Output    │
                    └─────────────┘     └─────────────┘
                          │
                          ▼
                    ┌─────────────┐
                    │  Retrieval  │◀──── Query dari user
                    │  (Cari)     │
                    └─────────────┘
                          │
                          ▼
                    ┌─────────────┐     ┌─────────────┐
                    │   Context   │────▶│    LLM      │────▶ Response
                    │  (Gabung)   │     │  (Generate) │
                    └─────────────┘     └─────────────┘
```

**Implementasi RAG Step-by-Step**

```python
# ============================================
# LANGKAH 1: Install dependencies
# ============================================
# pip install langchain openai chromadb tiktoken

# ============================================
# LANGKAH 2: Import libraries
# ============================================
from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma
from langchain.chat_models import ChatOpenAI
from langchain.chains import RetrievalQA

# ============================================
# LANGKAH 3: Load dokumen
# ============================================
loader = PyPDFLoader("document.pdf")
documents = loader.load()

# ============================================
# LANGKAH 4: Pecah dokumen menjadi chunks
# ============================================
# Mengapa perlu dipecah?
# - Context window terbatas
# - Lebih fokus untuk retrieval
# - Mengurangi noise

text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,      # Ukuran setiap chunk
    chunk_overlap=200,     # Overlap antar chunk untuk konteks
    length_function=len,   # Mengukur panjang dengan karakter
)

docs = text_splitter.split_documents(documents)

# ============================================
# LANGKAH 5: Buat embeddings dan simpan ke vector DB
# ============================================
# Embeddings mengubah teks menjadi VEKTOR (array angka)
# Teks yang semakna akan punya vektor yang mirip

embeddings = OpenAIEmbeddings()

# Simpan ke Chroma (vector database lokal)
vectorstore = Chroma.from_documents(
    documents=docs,
    embedding=embeddings,
    persist_directory="./chroma_db"  # Lokasi penyimpanan
)

# ============================================
# LANGKAH 6: Setup retrieval
# ============================================
# Retrieval = mencari chunks yang paling relevan dengan query

retriever = vectorstore.as_retriever(
    search_type="similarity",    # Cari berdasarkan kemiripan
    search_kwargs={"k": 3}       # Ambil 3 dokumen teratas
)

# ============================================
# LANGKAH 7: Setup LLM dan QA chain
# ============================================
llm = ChatOpenAI(
    model_name="gpt-4o",
    temperature=0.0  # Low temperature untuk factual QA
)

qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",  # Masukkan semua retrieved docs ke prompt
    retriever=retriever,
    return_source_documents=True
)

# ============================================
# LANGKAH 8: Query!
# ============================================
query = "Apa kesimpulan utama dari dokumen ini?"
result = qa_chain({"query": query})

print(result["result"])
print("\nSource documents:")
for doc in result["source_documents"]:
    print(f"- {doc.metadata.get('source', 'Unknown')}")
```

##### Breakdown Detail: RAG Architecture

**RAG Pipeline Stages:**

**Stage 1: Document Loading**
- [ ] Load documents dari various sources
- [ ] Supported formats: PDF, DOCX, TXT, HTML, Markdown, CSV
- [ ] Use LangChain document loaders:
  - `PyPDFLoader` untuk PDF
  - `UnstructuredURLLoader` untuk web pages
  - `Docx2txtLoader` untuk Word docs
  - `CSVLoader` untuk CSV files
- [ ] Preserve metadata (source, page number, etc.)

**Stage 2: Text Chunking**
- [ ] Split documents jadi smaller pieces
- [ ] **Chunk size**: Typically 500-1500 characters
- [ ] **Overlap**: 100-300 characters antar chunks
- [ ] **Strategies**:
  - `RecursiveCharacterTextSplitter`: Recursive splitting
  - `CharacterTextSplitter`: Split by character count
  - `TokenTextSplitter`: Split by token count (better for LLMs)
- [ ] Considerations:
  - Too small: Lost context
  - Too large: Exceed context window, poor retrieval
  - Maintain semantic coherence

**Stage 3: Embedding Generation**
- [ ] Convert text chunks → numerical vectors
- [ ] Embedding models:
  - **OpenAI**: `text-embedding-3-small`, `text-embedding-3-large`
  - **Google**: `embedding-001`, `text-embedding-preview`
  - **Open source**: `sentence-transformers` (all-MiniLM-L6-v2)
- [ ] Embedding dimensions: 384, 768, 1536, 3072
- [ ] Cost: Charged per 1K tokens

**Stage 4: Vector Storage**
- [ ] Store embeddings in vector database
- [ ] Each vector linked to original chunk
- [ ] Metadata indexing untuk filtering
- [ ] Persist untuk reuse

**Stage 5: Query Processing**
- [ ] User query → Embedding vector
- [ ] Same embedding model untuk consistency

**Stage 6: Retrieval**
- [ ] Similarity search antara query dan stored vectors
- [ ] **Similarity metrics**:
  - Cosine similarity: Most common
  - Euclidean distance: Direct distance
  - Dot product: Fast, good for normalized vectors
- [ ] Return top-K most similar chunks
- [ ] **K value**: Typically 3-10

**Stage 7: Augmentation**
- [ ] Combine retrieved context + user query
- [ ] Prompt template:
```
Context: {retrieved_chunks}
Question: {user_question}
Answer based on the context provided.
```
- [ ] Include source references

**Stage 8: Generation**
- [ ] Send augmented prompt ke LLM
- [ ] LLM generates answer based on context
- [ ] Optional: Force citation

**Advanced RAG Patterns:**

**1. Query Expansion:**
- [ ] HyDE (Hypothetical Document Embeddings)
- [ ] Multi-query retrieval
- [ ] Query decomposition

**2. Retrieval Enhancement:**
- [ ] Sentence window retrieval
- [ ] Parent document retriever
- [ ] Auto-merging retrieval
- [ ] Ensemble retrieval (multiple retrievers)

**3. Re-ranking:**
- [ ] Apply ML model untuk re-rank results
- [ ] Tools: Cohere Rerank, BGE-Reranker
- [ ] Cross-encoder vs bi-encoder

**4. Hybrid Search:**
- [ ] Combine dense (embedding) + sparse (BM25) retrieval
- [ ] Better than either alone
- [ ] Tools: Weaviate, Qdrant support hybrid

**5. Iterative RAG:**
- [ ] Generate, critique, refine
- [ ] Multi-step reasoning
- [ ] Self-correction loops

**6. Graph RAG:**
- [ ] Extract entities dan relationships
- [ ] Build knowledge graph
- [ ] Query graph untuk context

**7. Reranking & Filtering:**
- [ ] Remove irrelevant chunks
- [ ] Diversify retrieved results
- [ ] Max marginal relevance

**Evaluation Metrics:**
- [ ] **Retrieval Quality**:
  - Context Precision: Relevant docs ranked high
  - Context Recall: All relevant docs retrieved
- [ ] **Generation Quality**:
  - Faithfulness: Answer matches context
  - Answer Relevancy: Answer addresses question
  - Hallucination Rate: Facts from context

**Common RAG Pitfalls:**
- [ ] Poor chunking strategy
- [ ] Wrong embedding model
- [ ] Insufficient retrieval (K terlalu kecil)
- [ ] Irrelevant context included
- [ ] Not handling duplicates
- [ ] Ignoring metadata filtering

**RAG Optimization Tips:**
- [ ] Experiment dengan chunk sizes
- [ ] Use specialized embeddings (e.g., CODE-berta untuk code)
- [ ] Implement query expansion
- [ ] Add re-ranking step
- [ ] Use hybrid search
- [ ] Implement self-query for metadata filtering

**Tools Ecosystem:**
- [ ] **LangChain**: Integrations dengan 50+ vector DBs
- [ ] **LlamaIndex**: More RAG-focused abstractions
- [ ] **Haystack**: Open source, by deepset
- [ ] **Embedchain**: Simplified RAG framework

**Indikator Proficiency:**
- [ ] Bisa build basic RAG pipeline from scratch
- [ ] Paham semua stages dari loading ke generation

##### Breakdown Detail: Vector Databases

**What is a Vector Database?**
- [ ] Special database optimized untuk similarity search
- [ ] Stores high-dimensional vectors (embeddings)
- [ ] Enables fast nearest neighbor search
- [ ] Used for semantic search, recommendations, RAG

**Core Concepts:**

**1. Embeddings:**

- [ ] **Cosine Similarity**: Angle between vectors (0 to 1)
- [ ] **Dot Product**: Raw similarity (-inf to +inf)
- [ ] **Euclidean Distance**: Straight-line distance
- [ ] Cosine most common untuk text embeddings

# BAGIAN III: AI PROMPT ENGINEERING - ROADMAP JALUR CEPAT PAHAM

## 3.1 Definisi Prompt Engineering

### Apa Itu Prompt Engineering?

**Prompt Engineering** adalah seni dan ilmu dalam merancang **instruksi yang efektif** untuk AI agar menghasilkan output yang akurat, relevan, dan sesuai kebutuhan.

### Analogi Sederhana

Bayangkan AI seperti **asisten baru yang sangat pintar tapi buta konteks**:
- Sangat pintar dalam memproses informasi
- Tapi TIDAK tahu konteks pekerjaan Anda
- Tidak tahu preferensi Anda
- Tidak tahu constraint bisnis Anda

Prompt engineering adalah seni **menyampaikan maksud Anda** dengan cara yang bisa dipahami AI.

### Prompt Engineering vs Context Engineering vs Requirement Architecture

```
Evolusi Terminologi:

2023: "Prompt Engineering" - Fokus pada kata-kata dalam prompt
2024: "Context Engineering" - Fokus pada bagaimana memberikan konteks
2026: "Requirement Architecture" - Fokus pada merancang sistem instruksi

Perbedaannya:

Prompt Engineering:
"Write a Python function to calculate factorial"
↓
"Just write the function"
↓
Context Engineering:
"As a senior Python developer, write a function..."
↓
Butuh additional context untuk specificity
↓
Requirement Architecture:
- Task: Write factorial function
- Constraints: Must handle n=0 to n=1000
- Style: Follow PEP8
- Output: Return integer only
- Edge cases: Raise ValueError for negative
```

## 3.3 Framework Prompt - CRISP

Dari Panduan-AI-Programming.md, Anda sudah mengenal framework **CRISP**:

### CRISP Breakdown

```
C - CONTEXT (Konteks)
├── Situasi/latar belakang
├── Teknologi yang digunakan
├── Tujuan aplikasi
└── Kondisi saat ini

R - ROLE (Peran)
├── Peran AI yang diharapkan
├── Expertise level
└── Gaya penulisan

I - INSTRUCTION (Instruksi)
├── Perintah spesifik
├── Kata kerja aktif
└── Action yang jelas

S - SPECIFICATIONS (Spesifikasi)
├── Technical requirements
├── Batasan yang harus dipatuhi
└── Functional requirements

P - PURPOSE (Tujuan)
├── Mengapa ini penting
├── Hasil akhir yang diharapkan
└── Business context
```

### Contoh CRISP Prompt

```
[C - CONTEXT]
Saya sedang membuat aplikasi e-commerce menggunakan React dan Node.js.
Aplikasi ini menggunakan PostgreSQL sebagai database dan JWT untuk auth.

[R - ROLE]
Sebagai senior fullstack developer dengan 10 tahun pengalaman,

[I - INSTRUCTION]
tuliskan API endpoint untuk registrasi pengguna baru.

[S - SPECIFICATIONS]
- Endpoint: POST /api/users/register
- Terima JSON dengan field: email, password, name
- Email harus valid format dan unik
- Password minimal 8 karakter, harus ada uppercase dan number
- Simpan password sebagai hash (bcrypt)
- Return user object tanpa passwordfield

[P - PURPOSE]
Ini untuk fitur registrasi di aplikasi e-commerce saya.
User perlu bisa membuat akun sebelum bisa berbelanja.
```

## 3.8 Evaluation Metrics untuk Prompts

### The 6 Dimensions of Prompt Quality

| Dimensi | Bobot | Apa yang Diukur |
|---------|-------|-----------------|
| Clarity | 20% | Seberapa jelas instruksi? |
| Specificity | 15% | Seberapa detail spesifikasi? |
| Context | 15% | Apakah konteks cukup? |
| Few-Shot Examples | 10% | Apakah contoh membantu? |
| Role Prompting | 15% | Apakah role membantu? |
| TCOF Structure | 20% | Apakah Task, Context, Output, Format ada? |

### Scoring System

```python
# Sistem scoring sederhana untuk prompts

def score_prompt(prompt: str) -> dict:
    scores = {
        "clarity": 0,
        "specificity": 0,
        "context": 0,
        "few_shot": 0,
        "role": 0,
        "tcof": 0
    }
    
    # Check clarity - apakah ada instruksi yang ambigu?
    if "jelas" in prompt.lower() or "specific" in prompt.lower():
        scores["clarity"] = 1
    
    # Check specificity - apakah ada angka atau detail?
    import re
    if re.search(r'\d+', prompt):  # Ada angka
        scores["specificity"] = 1
    
    # ... dst untuk dimensi lain
    
    total = sum(scores.values()) / len(scores) * 100
    return {"dimensions": scores, "total_score": total}

# Usage
result = score_prompt(my_prompt)
print(f"Prompt Score: {result['total_score']:.1f}%")
```

## 4.1 Definisi Agentic AI

### Apa Itu Agentic AI?

**Agentic AI** adalah sistem AI yang bisa **berpikir, merencanakan, dan bertindak secara autonomous** untuk mencapai tujuan tertentu. Berbeda dengan AI biasa yang hanya merespons input, agentic AI bisa:

- Memecah tugas kompleks menjadi langkah-langkah
- Menggunakan tools untuk berinteraksi dengan dunia luar
- Beradaptasi dengan situasi yang berubah
- Mengevaluasi hasil dan mengoreksi diri sendiri

### Perbandingan: Traditional AI vs Agentic AI

```
TRADITIONAL AI:
User: "Apa cuaca besok?"
AI:  "Besok cerah dengan suhu 28°C"
```

### 1. Context - Apa yang Agent Tahu?

Context adalah **semua informasi** yang digunakan agent untuk membuat keputusan:

```
Types of Context:

IMMEDIATE CONTEXT:
├── User's current request
├── Previous conversation turns
└── Session state

RETRIEVED CONTEXT:
├── From vector database (RAG)
├── From knowledge graphs
└── From external APIs

MEMORY CONTEXT:
├── Short-term memory (current session)
├── Long-term memory (persistent)
└── Episodic memory (past experiences)
```

## 4.3 Agent Loop Patterns

### ReAct Pattern (Reasoning + Acting)

Ini adalah pattern paling fundamental untuk agentic systems:

```python
"""
ReAct = Reason + Act + Observe

LOOP:
1. REASON - Pikirin apa yang harus dilakukan
2. ACT - Lakukan action tertentu
3. OBSERVE - Lihat hasilnya
4. Loop sampai goal tercapai atau max iterations

Contoh: User bertanya tentang cuaca Bandung
"""

# Implementasi conceptual:
import json

class SimpleReActAgent:
    def __init__(self, llm, tools):
        self.llm = llm
        self.tools = tools
        self.max_iterations = 10
    
    def run(self, user_goal):
        # Initialize
        history = []
        observation = ""
        
        for i in range(self.max_iterations):
            # 1. REASON
            prompt = self.build_reasoning_prompt(user_goal, history, observation)
            reasoning = self.llm.generate(prompt)
            
            # Parse reasoning untuk extract next action
            # Contoh reasoning output:
            # {
            #   "thought": "User ingin tahu cuaca Bandung besok",
            #   "action": "web_search",
            #   "action_input": "weather Bandung Indonesia tomorrow"
            # }
            
            parsed = json.loads(reasoning)
            thought = parsed.get("thought")
            action = parsed.get("action")
            action_input = parsed.get("action_input")
            
            # Check if task is complete
            if action == "finish":
                return parsed.get("output")
            
            # 2. ACT
            if action in self.tools:
                result = self.tools[action].execute(action_input)
            else:
                result = f"Unknown action: {action}"
            
            # 3. OBSERVE
            observation = result
            
            # Append to history
            history.append({
                "thought": thought,
                "action": action,
                "action_input": action_input,
                "observation": observation
            })
        
        return "Max iterations reached"
    
    def build_reasoning_prompt(self, goal, history, observation):
        # Build prompt untuk reasoning step
        history_text = "\n".join([
            f"Step {i+1}: Thought: {h['thought']}, Action: {h['action']}, Observation: {h['observation']}"
            for i, h in enumerate(history)
        ])
        
        return f"""
Goal: {goal}

Previous steps:
{history_text}

Current observation: {observation}

Based on the above:
1. What should I do next?
2. Should I use a tool or respond directly?

Respond in JSON format:
{{
    "thought": "What am I thinking now?",
    "action": "tool_name or 'finish'",
    "action_input": "input for the tool (if applicable)",
    "output": "final answer (if finishing)"
}}

Available tools: {list(self.tools.keys())}
"""
```

### Plan-and-Execute Pattern

Berbeda dengan ReAct yang interleaves planning dan acting, Plan-and-Execute mendahului semua planning:

```python
"""
Plan-and-Execute Pattern:

1. PLANNER - Buat plan lengkap di awal
2. EXECUTOR - Eksekusi plan step by step
3. Bisa replan jika ada perubahan

LOOP:
[PLAN] ──▶ [EXECUTE 1] ──▶ [EXECUTE 2] ──▶ ...
             │                │
             ▼                ▼
          [REPLAN?]       [REPLAN?]
             │                │
             ▼                ▼
           Yes              Yes
             │                │
             ▼                ▼
         [NEW PLAN]      [NEW PLAN]
"""

class PlanAndExecuteAgent:
    def __init__(self, planner_llm, executor_llm, tools):
        self.planner = planner_llm
        self.executor = executor_llm
        self.tools = tools
    
    def plan(self, goal):
        # Planner membuat plan lengkap
        prompt = f"""
Goal: {goal}

Break down this goal into clear, sequential steps.
Consider:
- Prerequisites for each step
- Potential dependencies
- Tools that might be needed

Output format:
Step 1: [description]
Step 2: [description]
...

Be specific but flexible - plan can be adjusted during execution.
"""
        plan_text = self.planner.generate(prompt)
        return self.parse_plan(plan_text)
    
    def execute_plan(self, plan):
        results = []
        for step in plan["steps"]:
            # Execute each step
            result = self.execute_step(step, results)
            results.append({"step": step, "result": result})
            
            # Check if replanning is needed
            if self.should_replan(step, result):
                # Replan with updated context
                new_plan = self.replan(goal, results)
                # Continue with new plan
                plan = new_plan
        
        return self.synthesize(results)
    
    def execute_step(self, step, previous_results):
        # Execute dengan context dari hasil sebelumnya
        prompt = f"""
Task: {step}

Previous results that might be relevant:
{previous_results}

Execute this task and provide the result.
"""
        return self.executor.generate(prompt)
```

### Tool-Augmented Planning

Agent yang menggunakan tool untuk membantu planning:

```python
"""
Tool-Augmented Planning:

Agent TIDAK hanya planning dengan LLM reasoning,
tapi menggunakan tools untuk GATHER INFORMATION
sebelum planning.

LOOP:
1. Analyze goal
2. Identify information gaps
3. Use tools to fill gaps
4. Create/update plan
5. Execute
6. Repeat
"""

class ToolAugmentedPlanner:
    def __init__(self, llm, tools, research_tool):
        self.llm = llm
        self.tools = tools
        self.research = research_tool
    
    def run(self, goal):
        plan = {"steps": [], "status": "initial"}
        
        while plan["status"] != "complete":
            # Analyze - apa yang kita tahu dan tidak tahu?
            analysis = self.analyze_goal_and_plan(goal, plan)
            
            # If there are information gaps, research
            if analysis["gaps"]:
                for gap in analysis["gaps"]:
                    info = self.research.query(gap)
                    plan["information"].append(info)
                    plan = self.update_plan(plan, info)
            
            # If plan is ready, execute next step
            if not analysis["gaps"] and plan["steps"]:
                next_step = plan["steps"].pop(0)
                result = self.execute_step(next_step)
                plan["completed_steps"].append({"step": next_step, "result": result})
                
                # Check if done
                if self.is_goal_complete(goal, plan["completed_steps"]):
                    plan["status"] = "complete"
        
        return self.format_output(plan)
```

```
│  └── Stored in: Vector DB, SQL, or file                     │

│  EPISODIC MEMORY                                            │
```

```python
# Contoh Memory Implementation

from langchain.memory import ConversationBufferMemory
from langchain.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings

class AgentMemory:
    def __init__(self):
        # Short-term: conversation buffer
        self.stm = ConversationBufferMemory(
            memory_key="chat_history",
            return_messages=True,
            output_key="output"
        )
        
        # Long-term: vector store
        embeddings = OpenAIEmbeddings()
        self.ltm = Chroma(
            persist_directory="./memory_db",
            embedding_function=embeddings
        )
    
    def add_interaction(self, user_input, agent_response):
        # Add to short-term
        self.stm.save_context(
            {"input": user_input},
            {"output": agent_response}
        )
        
        # Add to long-term (if important)
        if self.is_important(user_input, agent_response):
            self.ltm.add_texts([
                f"User preference: {user_input}",
                f"Agent response: {agent_response}"
            ])
    
    def retrieve_relevant(self, query, k=5):
        # Retrieve from long-term
        docs = self.ltm.similarity_search(query, k=k)
        return docs
    
    def get_conversation_summary(self):
        # Get summary of recent conversation
        return self.stm.load_memory_variables({})
```

```python
import json

class ToolRegistry:
    def __init__(self):
        self.tools = {}
    
    def register(self, name, func, description, parameters):
        self.tools[name] = {
            "function": func,
            "description": description,
            "parameters": parameters
        }
    
    def execute(self, name, arguments):
        if name not in self.tools:
            return {"error": f"Tool '{name}' not found"}
        
        try:
            args = json.loads(arguments) if isinstance(arguments, str) else arguments
            result = self.tools[name]["function"](**args)
            return {"success": True, "result": result}
        except Exception as e:
            return {"success": False, "error": str(e)}
```

## 4.7 Safety, Guardrails, dan Evaluation

### Safety Considerations

```python
"""
Agent Safety Checklist:

1. INPUT VALIDATION
   - Sanitize user input sebelum diproses
   - Validasi format dan type
   - Batasi panjang input

2. OUTPUT FILTERING
   - Cek output sebelum return ke user
   - Filter informasi sensitif
   - Validate format output

3. ACTION APPROVAL
   - Human-in-the-loop untuk actions kritis
   - Confirmation untuk destructive actions
   - Rate limiting untuk prevent abuse

4. RESOURCE LIMITS
   - Max iterations
   - Max token usage
   - Timeout untuk long operations

5. ERROR HANDLING
   - Graceful degradation
   - Meaningful error messages
   - Fallback strategies
"""

class AgentSafetyGuardrails:
    def __init__(self, agent):
        self.agent = agent
        self.max_iterations = 10
        self.max_tokens = 4000
        self.critical_actions = ["delete", "send_email", "payment"]
    
    def pre_process_input(self, user_input):
        # Sanitize input
        sanitized = user_input.strip()
        
        # Length check
        if len(sanitized) > 10000:
            raise ValueError("Input terlalu panjang")
        
        return sanitized
    
    def pre_process_action(self, action, action_input):
        # Check for critical actions
        if action in self.critical_actions:
            # Require human approval
            if not self.get_human_approval(action, action_input):
                return {"approved": False, "reason": "Ditolak oleh user"}
        
        # Check action is in allowed list
        if action not in self.agent.allowed_tools:
            return {"approved": False, "reason": "Tool tidak diizinkan"}
        
        return {"approved": True}
    
    def post_process_output(self, output):
        # Filter sensitive information
        filtered = self.filter_sensitive(output)
        
        # Validate format
        if not self.validate_output_format(filtered):
            return {"error": "Output format tidak valid"}
        
        return {"output": filtered}
    
    def filter_sensitive(self, text):
        # Remove potential sensitive patterns
        import re
        patterns = [
            r'\b\d{16}\b',  # Credit card
            r'\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b',  # Email
            r'\b\d{13,16}\b',  # Phone numbers
        ]
        
        for pattern in patterns:
            text = re.sub(pattern, '[REDACTED]', text)
        
        return text
```

```python
    def generate_evaluation_report(self, agent_run):
```
